### `219-feature-ribbon-part.patch` 维护记录

- 2026-07-19：彻底修复全高纵向按钮（`ribbon-item-full` + `ribbon-item-vertical`）的 Codicon 图标过小问题。前两次修复用 `.ribbon-root .ribbon-item-icon.codicon`（特异性 0,3,0）尝试覆盖 Codicon 固定 16px 字号，但打包后的 `workbench.desktop.main.css` 里存在一条更狠的上游规则 `.monaco-workbench .part .codicon[class*='codicon-'] { font-size: calc(var(--vscode-workbench-font-size) * 1.230769) }`（特异性 0,4,0），它把字号锁回 13px；规则在 `src/vs` 源码中搜不到，疑似由 esbuild postprocess 或运行时 zoom 处理注入，但只要它存在于打包产物中就会压制 0,3,0 覆盖。修复方案是把覆盖选择器锚到 `.monaco-workbench .part.ribbon .ribbon-root .ribbon-item-icon.codicon` 与 `.monaco-workbench .ribbon-overlay .ribbon-item-icon.codicon`（特异性 0,5,0，针对折叠浮层用 `.monaco-workbench` 前缀而非 `.part`，因为 ContextView 容器位于 Workbench 直接子级而非 Part 内），同时给 `ribbon-group-dropdown-icon` 提供等价覆盖。完整 workbench CSS 环境下，icon 实际 `font-size` 从 13px 提升到 55px，图标高度占按钮高度从约 18% 提升到约 74%。
- 2026-07-19：降低内置 Ribbon tab 优先级。`BUILTIN_TABS` 的 `order` 从 `10 / 20 / 100` 调至 `1000 / 2000 / 3000`，扩展声明的 tab（默认 `order=0`）会自然排在内置 `File / View / Help` 之前；同步把 `ensureActiveTab` 的 fallback 从硬编码 `File` 改为排序后第一个 tab，使启动激活的 tab 与视觉首位一致。内置 tab 当前仅作为 Ribbon 实现的测试占位，正式版将移除。
- 2026-07-19：内置 Ribbon tab 与内置菜单分组标题按当前界面语言显示；中文语言环境下 `File / View / Help` 与对应分组标题显示中文，非中文语言环境保留 VS Code 常规本地化结果。
- 2026-07-19：去掉 `localizeRibbonLabel` 包装函数，改为模块顶层 `isZhHansLocale` 常量配合 `isZhHansLocale ? "中文" : localize('key', "Default")` 内联三元。原因：上游 `build/lib/nls-analysis.ts` 的 `parseLocalizeKeyOrValue` 会对源码里每个 `localize(...)` 的第一参数做 `eval`，第一参数必须是字面量；而包装函数里的 `localize(key, defaultValue)` 第一参数是变量 `key`，eval 时抛 `ReferenceError: key is not defined`，导致发行 CI 在 `vscode-min-prepack` 阶段失败。本地 `prepare.sh` 不跑 min-prepack，所以 CI 才暴露。
- 2026-07-20：修复占用 ribbon 面板全高的两类按钮（「上 icon + 下 标题」= `ribbon-item-full` + `ribbon-item-vertical`，以及「icon + 标题 + 下拉按钮」= `ribbon-dropdown-item` 系列，含 splitButton 主按钮/trigger 与 dropdown-single footer）的三个问题。① 图标过大：把 `.ribbon-item-vertical .ribbon-item-icon`、`.ribbon-dropdown-primary .ribbon-item-icon`、`.ribbon-dropdown-single > .ribbon-item-icon` 的 `--ribbon-icon-size` 由 `calc(var(--ribbon-item-full-height) - 19px / 22px)`（实际 52–55px）统一改为固定 40px，并去掉 `flex: 1 1 0` 拉伸，改为 `flex: 0 0 auto` 固定 40×40 盒子。② 内容贴底：原 vertical 按钮因 icon 拉伸把 label 顶到面板底、dropdown 因 primary `flex: 1 1 0` 把 trigger 推到底；改为 `.ribbon-item-vertical` 加 `justify-content: flex-start`、`.ribbon-dropdown-item` 加 `justify-content: flex-start`、primary/single 改 `flex: 0 0 auto`，让 icon + 标题从顶部开始排列、底部留白。③ chevron-down 水平对齐：因不同按钮标题 1 行/2 行混排，trigger/footer 默认 `flex-direction: row` 让 chevron 垂直位置参差；新增 `watchRibbonLabelLines()` 私有方法，在 trigger/footer 渲染后用 `ResizeObserver` + window `resize` 监听 `.ribbon-item-label` 的 `offsetHeight`，超过单行行高（>15px）即判定为 2 行，给容器 toggle `ribbon-label-single` / `ribbon-label-double` class：`.ribbon-label-single` 让 chevron 排在标题下方（`flex-direction: column`），`.ribbon-label-double` 保持标题右侧（`flex-direction: row; align-items: flex-end`），两种情况下 chevron 都落在按钮底部附近，保证整排水平对齐。`createDropdownTrigger` 同步新增 `store: DisposableStore` 参数以托管监听器生命周期。
- 2026-07-20：修复折叠分组下拉菜单的「切换不弹」问题并新增 Word 风格面板下拉。① 切换不弹：原实现里折叠分组下拉（内置 tab 走 `IContextMenuService`、扩展 tab 走 `IContextViewService`）只有单向 show，没有 open 状态跟踪；当 `selectTab`/`render`/`renderActiveContent`/`refreshActiveContent` 重建 DOM 时会顺手清掉 anchor 元素，上游 `ContextView` 单例检测到 anchor 失联后静默 hide 旧菜单，但新点击事件绑在已销毁的旧 DOM 上，导致「旧的收了、新的没弹」。修复方案：给 `RibbonPart` 加 `openMenuAnchor` / `openMenuId` / `openMenuKind`（`'contextView' | 'contextMenu'`）三字段，所有下拉 CLICK 入口先做 toggle 判定（同 anchor → `closeOpenMenu()` 返回），否则 `closeOpenMenu()` 后再 show 并记录新状态；`render`/`renderActiveContent`/`refreshActiveContent`/`showOverlay` 入口主动 `closeOpenMenu()`，避免重建 DOM 与菜单显示抢跑。覆盖三种路径：折叠分组之间、切 tab、点 ribbon 上任意普通按钮。内置 tab 的 `renderGroupDropdown` 与扩展 tab 的 `renderContributionGroupDropdown`、splitButton trigger、dropdown-single 按钮均打通此机制；扩展下拉 anchor 上打 `data-menu-id` 用于 `showContributionMenu` 的 toggle 判定。② 面板下拉：新增 `RibbonMenuLayout = 'menu' | 'panel'` 与 `IRibbonMenuContribution.{layout,title,columns}`，`showContributionMenu` 在 `layout==='panel'` 时分流到新方法 `showPanelDropdown`。`showPanelDropdown` 同时作为 `IRibbonService` 公共方法暴露（返回 `IRibbonPanelHandle`），扩展运行时可直接调用而不必自建 HTML。DOM 骨架 `.ribbon-panel-dropdown` → `.ribbon-panel-header`（可选标题 + 关闭按钮）→ `.ribbon-panel-body` → 多个 `.ribbon-panel-section`（可选小标题 + `.ribbon-panel-grid` 按 `--ribbon-panel-columns` 列网格）+ 可选 `iframe.ribbon-custom-content`。CSS 使用专用色板 `--vscode-ribbon-dropdown-*`（在 `.monaco-workbench` 上 fallback 到 `editorHoverWidget-*` / `list-*`），不依赖 `--vscode-menu-*`，避免观感像菜单栏。`renderContributionItem` 生成的 `.ribbon-item` 在 `.ribbon-panel-cell` 语境下被 CSS 改写成「28px 大图标在上、11px label 居中」的方形格子。`ribbonService.ts` 同步新增 `IRibbonPanelItem` / `IRibbonPanelSection` / `IRibbonPanelDescriptor` / `IRibbonPanelHandle` 类型，并对声明式场景通过 `renderItem` 回调 + `as IRibbonItemContribution` 断言复用 `renderContributionItem`。内置 tab（File/View/Help）的折叠下拉按用户要求保持 `IContextMenuService` 实现不动。
- 2026-07-20：重做折叠分组下拉，统一改为「复用 `.ribbon-items` 原样 DOM」。原实现里扩展分组下拉走 `showContributionMenu` 默认分支（竖排 `.ribbon-menu-items`），内置分组下拉走 `IContextMenuService` 原生右键菜单，两者都不是该分组未折叠时在 ribbon 主区域里的样貌（横向 6 行网格、full/half/third 三种尺寸、splitButton 双层结构）。更糟的是 `renderContributionGroupDropdown` 第 1869 行 `contribution.menu?.layout` 是无效读取——`IRibbonGroupContribution` 接口根本没有 `menu` 字段，所以扩展分组下拉永远走竖排菜单分支，`layout:'panel'` 在分组级连声明入口都没有。新方案：新增私有方法 `showFoldedGroupDropdown(anchor, group, title)`，点击折叠分组下拉触发按钮时：① 走 toggle 判定（沿用 `openMenuAnchor`）；② 在原分组 `.ribbon-items` 节点前插入一个 `<template class="ribbon-items-placeholder">` 占位（不可见、不参与布局、不被 `:scope > .ribbon-items` 误命中）；③ 用 `contextViewService.showContextView` 弹出，`render(container)` 里给容器加 `ribbon-panel-dropdown` + `ribbon-folded-dropdown` 双 class（复用 PR 已有 `--vscode-ribbon-dropdown-*` 外壳色板），显式 `container.style.setProperty('--ribbon-item-full-height', '74px')`（弹层不在 `.ribbon-group` 子树里继承不到该变量，不设会导致网格高度塌缩），然后 `container.appendChild(items)` 把原节点借过来；④ `onHide` 里 `placeholder.parentNode.insertBefore(items, placeholder)` 还回原位再 `placeholder.remove()`，若 `placeholder.parentNode` 为 null（说明 ribbon 已重建 content 把原分组 DOM 销毁了）则让节点随 ContextView 容器一并清理。`renderContributionGroupDropdown` 与 `renderGroupDropdown` 的 CLICK 都改为调 `showFoldedGroupDropdown`，方法签名参数名从 `parent` 改为 `group`（调用方本来传的就是 group 元素），内置 tab 不再走 `contextMenuService.showContextMenu`、`openMenuKind='contextMenu'` 分支保留给可能的 item 级使用。CSS 新增 `.ribbon-folded-dropdown { padding: 6px }` 与 `.ribbon-folded-dropdown > .ribbon-items { height: var(--ribbon-item-full-height, 74px) }`，不覆写 `.ribbon-item` / `.ribbon-dropdown-item` 尺寸规则——复用 DOM 时原 ribbon 区域 CSS 照常生效，这正是「弹出原样内容」的核心收益。`.ribbon-group.folded .ribbon-items { display: none }` 的选择器特异性（0,3,0）只命中 `.ribbon-group.folded` 后代里的 `.ribbon-items`，弹层容器挂在 ContextView 根下不在该子树里，`display:none` 不会命中弹层中的节点。时序安全：`render`/`renderActiveContent` 已是 `closeOpenMenu()` → `clearNode`，onHide 先于 clearNode 触发把节点还回。`layout:'panel'` 仅对按钮级（splitButton/dropdown 的 `menu`）生效，分组级无视，与用户需求一致。

## Ribbon 内部 API 登记

| 内部 API / 耦合点 | 上游位置 | 用途 |
| --- | --- | --- |
| `Part` / `SerializableGrid` | `src/vs/workbench/browser/part.ts`、`browser/layout.ts` | 注册全宽 Ribbon Part 并参与 Workbench 布局 |
| `ExtensionsRegistry` | `services/extensions/common/extensionsRegistry.ts` | 注册并校验 `contributes.ribbon` |
| `IContextViewService` | `platform/contextview/browser/contextView.ts` | 显示 Ribbon 菜单弹层、面板下拉、折叠分组下拉（`showFoldedGroupDropdown`）与折叠状态临时覆盖层；递归菜单、面板与折叠分组下拉共用同一 Context View 单例（故需要 `closeOpenMenu` 做切换/toggle 状态管理） |
| `IContextMenuService` | `platform/contextview/browser/contextView.ts` | 内置 action 的 `SubmenuAction` 子菜单弹出（`renderActionGroup` 第 519 行）；折叠分组下拉已不再走此服务，统一改为复用 `.ribbon-items` 原样 DOM |
| `IRibbonService.showPanelDropdown` | `browser/parts/ribbon/ribbonService.ts` + `ribbonPart.ts` | 扩展运行时调用的 Word 风格面板下拉公共 API；返回 `IRibbonPanelHandle` 供主动关闭 |
| `IPartsSplash` | `workbench/contrib/splash/browser/partsSplash.ts` | 在启动 Splash 中保留 Ribbon 高度 |

修改以上任一补丁时，必须同时验证干净 VS Code 基线和 VSCodium 官方补丁后的发行基线。
